# AI Agent 自動化驗收測試導入計畫書 (POC 階段)

## 一、 專案目標

導入基於大型語言模型（LLM）的 Agent 測試框架，驗證其是否能取代傳統人工，執行高重複性、枯燥的 Web E2E（端到端）使用者驗收測試。

- **短期目標：** 證明 AI 能根據自然語言腳本，成功操作瀏覽器並給出正確的測試判定。
- **長期願景：** 將測試流程容器化並整合進現有工作流，實現「人類寫劇本，AI 當測試員」的無程式碼測試生態。

## 二、 技術選型

- **核心框架：** Python 3.x
- **AI 代理與瀏覽器控制：** `browser-use` 框架（底層封裝 Playwright）
- **大語言模型 (LLM)：** Gemini 1.5 Pro（負責視覺 DOM 樹解析、步驟推理與斷言判定）
- **環境部署 (預定)：** Docker（確保執行環境乾淨隔離）與 GitLab CI（供後續自動化觸發）

---

## 三、 實作三階段 (Timeline: 預估 1-2 週)

### Phase 1：基礎環境與單點驗證 (Day 1-2)

**目標：** 打通「劇本輸入 -> AI 執行 -> 截圖存檔 -> 結果輸出」的最短路徑。

- 安裝 Python 環境與相關套件。
- 挑選 1 到 2 個**不需要複雜登入、狀態單純**的功能進行測試（例如：公開的知識庫搜尋、Issue Tracker 的公開頁面瀏覽）。
- 實作「每步操作強制截圖（Step-by-step screenshots）」機制，建立人類覆核的基礎。

### Phase 2：複雜場景攻防與提示詞工程 (Day 3-5)

**目標：** 挑戰真實世界的高頻操作場景，調優 AI 穩定度。

- **身分驗證測試：** 讓 AI 執行包含填寫帳號密碼、送出表單並處理登入 Session 的完整流程。
- **動態元件測試：** 測試 AI 是否能正確等待並處理 AJAX 異步載入的資料或彈出視窗（Modal）。
- **Prompt 優化：** 針對 AI 點錯按鈕或誤判的狀況，修正自然語言測試腳本的描述方式，找出最適合 AI 理解的「SOP 語法」。

### Phase 3：報告產出與 CI/CD 整合評估 (Day 6-7)

**目標：** 讓腳本具備工程化、可重複利用的價值。

- 將 Python 測試環境與腳本打包成 Docker Image，解決本機環境差異問題。
- 設計簡單的測試報告格式（例如將測試結果與截圖路徑輸出為 Markdown 檔案）。
- 評估如何透過 Webhook 將測試結果自動拋轉（例如推送到通訊軟體或整合回內部看板）。

## 四、 POC 階段的評估指標 (KPIs)

實驗結束後，我們將透過以下指標決定是否擴大導入：

1. **腳本建置成本：** 撰寫一份 AI 自然語言測試劇本，對比過去手寫 Cypress/Selenium 腳本的時間差異。
2. **執行穩定率 (Pass Rate)：** 在網頁介面不變的情況下，AI 能否連續 5 次精準執行相同的任務而不發生「幻覺」或點錯位置。
3. **防禦網頁改版的能力：** 刻意微調網頁的 UI（例如改變按鈕顏色或更換 DOM 層級），測試 AI 是否仍能透過視覺辨識正確完成任務。

## 五、 潛在風險與對策

- **風險：AI 行為不穩定 (Flakiness)**
- **對策：** 嚴格要求 AI 產出執行日誌與步驟截圖。當發生錯誤時，由人類開發者扮演「測試導演」來觀看截圖，快速釐清是系統真有 Bug 還是 AI 迷路。

- **風險：單次執行成本與速度較傳統腳本高**
- **對策：** 策略性分工。這套 AI 測試不取代毫秒級的單元測試（Unit Test），僅專注於取代**最具商業價值、且人類點擊最痛苦的關鍵 E2E 流程**。
