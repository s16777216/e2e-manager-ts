## Context

目前的系統雖已實作後端的 `PATCH /api/projects/:id` 與 `DELETE /api/projects/:id` API，但前端僅提供建立專案的功能，尚未整合專案資訊的編輯與刪除。此設計文件將規劃如何透過前端 UI/UX 的配合，以及 API 封裝與 Hook 的擴充，在不更動後端核心邏輯的前提下，提供專案名稱、描述的修改，以及安全的專案刪除功能。

## Goals / Non-Goals

**Goals:**
- 提供專案資訊（名稱、描述）的編輯 Dialog 元件。
- 在專案詳細頁提供直觀的編輯入口，並確保修改後頁面名稱與麵包屑皆能即時重繪更新。
- 整合「刪除專案」功能，並設計輸入專案名稱確認的二次驗證，避免使用者誤刪資料。
- 補齊前端 API 封裝與專案 Hook 的更新及刪除函數。

**Non-Goals:**
- 本次設計不涉及後端 API 的調整、路由路徑變更或資料庫 Table Schema 的更動。
- 不提供專案列表中直接編輯/刪除專案的 Actions 行，所有編輯操作皆集中於專案詳情頁中。

## Decisions

### 1. 編輯與刪除專案的功能入口
- **決策**：在專案詳情頁標題（`activeProject.name`）右側放置一個小型的筆型按鈕（`Edit2` 圖標），點擊後彈出 `EditProjectDialog`。
- **考量**：若在專案列表（Table）中新增 Actions 欄位，會佔用表格寬度，且專案管理並非高頻率操作，放在詳情頁中能保持主視覺的簡潔。

### 2. 專案刪除的二次確認機制
- **決策**：在 `EditProjectDialog` 中點擊「刪除專案」時，於 Dialog 內部將狀態切換至確認畫面（或是彈出第二層子對話框），並提供一個 Input 讓使用者手動輸入專案名稱。只有在輸入字串與當前專案名稱完全一致時，「確定刪除」按鈕才會啟用。
- **考量**：刪除專案在資料庫層級會以 `ON DELETE CASCADE` 關聯刪除所有 Group、Testcase 和 TestRun 紀錄。為了防止意外點選，強制的名稱比對是極為必要且常見的最佳實踐。

## Risks / Trade-offs

- **[Risk]** 刪除專案將導致大量歷史執行數據與測試步驟遺失，且無法復原。
  - **[Mitigation]** 在刪除確認對話框中使用醒目的紅色警告字體（Tailwind `text-destructive`），並強制進行專案名稱文字精確比對以完成刪除確認。
